# 📊 System Design: Facebook-Scale Capacity Planning

## 📝 Overview
A foundational guide to the art of **Capacity Planning** for hyper-scale systems. It focuses on the back-of-the-envelope calculations required to estimate storage, memory, and compute resources for platforms serving billions of daily active users, ensuring the infrastructure can handle both normal and peak loads.

!!! abstract "Core Concepts"

    - **Back-of-the-Envelope Estimation:** Quick, rough calculations to determine if a system design is feasible within physical constraints.
    - **80/20 Rule:** Assuming 80% of traffic is generated by 20% of data, allowing for efficient cache sizing.
    - **Safety Margins:** Applying replication factors and buffer percentages to account for failures and unexpected spikes.

## 🛠️ Functional & Non-Functional Requirements
### Functional Requirements

1. **Storage Planning:** Accurately estimate disk space for user-generated content over a 5-year horizon.
2. **Memory Planning:** Size distributed cache clusters based on access patterns and data lifespan.
3. **Compute Planning:** Determine the fleet size for application and worker nodes based on throughput and latency goals.

### Non-Functional Requirements

1. **Cost Efficiency:** Avoid massive over-provisioning while maintaining a performance buffer.
2. **Predictability:** Estimates should align with the expected growth of the user base.
3. **Accuracy:** Calculations must be within an order of magnitude of actual production requirements.

## 🧠 The Engineering Story

**The Villain:** "The Under-Provisioned Outage." Launching a viral feature only to have the database crash in 5 minutes because the storage was not estimated correctly.

**The Hero:** "The Capacity Planner." Using rough math to predict resource needs before a single line of code is written.

**The Plot:**

1. Estimate DAU and Read/Write ratios to understand the system's character.
2. Calculate Storage, Memory, and Network bandwidth requirements.
3. Apply replication factors (e.g., 3x) and safety margins for high availability.

**The Twist (Failure):** "The Long Tail." 99.9% of your data is rarely accessed, but the 0.1% "hot" data is so large it still exceeds your entire cache capacity.

**Interview Signal:** Mastery of **System Scaling Math**, **Resource Allocation**, and **Infrastructure Cost Estimation**.

## 🏗️ High-Level Architecture
Perform a back-of-the-envelope estimation for a platform with 2 Billion Daily Active Users (DAU), accounting for massive media uploads, real-time metadata access, and peak request traffic.

### Key Design Challenges:
#### 1. Storage Computation

- **Scenario:** Users upload 500M photos/day. Average photo size = 200KB.
- **Task:** Calculate total storage per day and per year (including 3x replication).
- **Math:** 500M photos * 200KB = 100TB/day. With 3x replication = 300TB/day. Yearly = ~110PB.

#### 2. RAM Estimation

- **Scenario:** We want to cache 20% of the daily metadata read requests.
- **Task:** Estimate total RAM needed for the Redis cluster.
- **Math:** If total daily metadata is 10TB, we need 2TB of RAM for a 20% cache hit ratio.

#### 3. Server Count

- **Scenario:** Each web server can handle 5,000 concurrent requests. Peak traffic is 500,000 requests/sec.
- **Task:** Estimate no. of web servers needed.
- **Math:** 500k / 5k = 100 servers (minimum). In practice, we add 20-30% overhead for safety = ~130 servers.
